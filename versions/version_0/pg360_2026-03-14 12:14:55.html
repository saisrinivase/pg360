HOT Updates, Fillfactor & Write Amplification
HOT (Heap-Only Tuple) updates avoid index churn. Low HOT ratio + default fillfactor = unnecessary WAL and vacuum overhead.
Why HOT Updates Matter: PostgreSQL's Most Underutilized Optimization EDUCATION
HOT Update: When an UPDATE modifies a row on the same page AND no indexed column changes, PostgreSQL can do a "Heap Only Tuple" update updating just the heap page with no index changes. This is 5-10x cheaper than a normal update: no index entry added, no WAL for index pages.
FILLFACTOR controls how full heap pages are packed: default is 100% (no free space). If you set FILLFACTOR=70, 30% of each page stays free for HOT updates on that same page.
When to set FILLFACTOR < 100: Tables with frequent UPDATEs to non-indexed columns. Cost: ~30% more storage. Benefit: 5-10x faster updates, less WAL, less vacuum pressure, better concurrency.

Planner Statistics Quality & Estimation Errors
Stale statistics cause wrong query plans. This section identifies tables where the planner is likely making bad row estimates.
Why This Matters: A Bad Estimate = A Catastrophically Wrong Plan PLAN QUALITY
The #1 cause of slow queries in PostgreSQL is bad planner row estimates.
If the planner thinks a table has 100 rows but it actually has 1,000,000 rows, it will: (1) Choose a Nested Loop join instead of a Hash Join 10,000x slower. (2) Not use an index it should use. (3) Not allocate enough work_mem for sorts.

Column Correlation Gaps
Multi-column WHERE clauses on correlated columns produce wrong estimates without extended stats PLAN QUALITY
Problem: When you write WHERE city='NYC' AND state='NY', the planner multiplies the individual selectivities: 5% 10% = 0.5%. But city and state are correlated! The actual selectivity might be 8%. This mis-estimate causes nested loops instead of hash joins.
Oracle equivalent: Oracle's adaptive query optimization handles this automatically. PostgreSQL requires manual CREATE STATISTICS.
Fix: CREATE STATISTICS stat_city_state ON city, state FROM addresses;
Then: ANALYZE addresses;
Verify: EXPLAIN shows actual vs estimated rows estimate should be within 2x of actual.

Autovacuum Full Advisor
Per-table vacuum thresholds, cost delay impact, wraparound countdown, and per-table tuning scripts.
Autovacuum is the Most Misunderstood and Most Critical PostgreSQL Feature CRITICAL KNOWLEDGE
What autovacuum does: (1) Reclaims dead tuple space. (2) Prevents XID wraparound. (3) Updates statistics for the planner. Without it: table bloat, planner failures, database shutdown.
The #1 mistake: The default vacuum threshold is 20% dead rows (autovacuum_vacuum_scale_factor=0.2). On a 100M row table, vacuum only triggers after 20M dead rows accumulate. By then your table is 30% larger, sequential scans are slower, and hot data is being evicted from cache.
The fix: Set per-table storage parameters: ALTER TABLE big_table SET (autovacuum_vacuum_scale_factor=0.01, autovacuum_analyze_scale_factor=0.005); This triggers vacuum after 1% dead rows (1M rows) instead of 20% (20M rows).
Cost delay: autovacuum_vacuum_cost_delay throttles vacuum speed to avoid I/O saturation. On modern SSDs, set to 2ms (from default 20ms) or 0 for very busy tables.

XID Wraparound Countdown (Database Level)
XID wraparound will cause PostgreSQL to refuse all writes and initiate emergency shutdown OUTAGE RISK
How it works: PostgreSQL uses 32-bit transaction IDs (XIDs). After ~2.1 billion transactions, XIDs wrap around. Old data appears to be "in the future" and becomes invisible (data loss simulation). PostgreSQL will shut down at 40M XIDs before wrap to prevent this.
Warning levels: age > 1.5B = CRITICAL (start emergency VACUUM FREEZE). age > 1B = HIGH (schedule VACUUM FREEZE this week). age > 500M = MEDIUM (monitor).
Emergency fix: VACUUM FREEZE VERBOSE schema.large_table;
Full database emergency: vacuumdb --all --freeze --analyze-in-stages
Prevention: Ensure autovacuum is running and not falling behind. Monitor with this report weekly.

Index Bloat Estimation
BTree index bloat from deletes and updates. Bloated indexes waste cache and slow all queries on those tables.
Index Bloat: Invisible Tax on Every Query STORAGE & PERF
Root Cause: When rows are deleted or updated, the index entries become "dead pages." BTree indexes reuse space for same-key inserts but not for different keys. On write-heavy tables, indexes can become 50-300% larger than needed.
Impact: Bloated indexes take more cache space, require more I/O to scan, and make every query on that table slower. An index with 60% bloat is literally 2.5x slower.
Fix: REINDEX CONCURRENTLY index_name; (PG12+ non-blocking)
Prevention: Set appropriate FILLFACTOR on high-churn indexes: CREATE INDEX idx ON table(col) WITH (fillfactor=70);

Row-Level Security Policy Coverage
Tables with sensitive data but no RLS any authenticated user can read all rows DATA ISOLATION
Root Cause: PostgreSQL tables are accessible to any role with SELECT privilege. Without RLS, user A can query user B's private data.

ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

GRANT SELECT ON orders TO app_role;
